# Product Content Audit — Architecture & State

**Last updated:** 2026-03-04
**Live URL:** https://product-content-audit.onrender.com
**Repo:** https://github.com/kyle11121/product-content-audit
**Hosting:** Render (Starter tier, auto-deploys on push to `main`)

---

## Purpose

Sales lead-gen tool for Pivotree. Shows B2B manufacturers how their product content "survives translation" across distributor websites. Audits live distributor product pages against the manufacturer's own PDP and scores content completeness across structured fields.

**Core principle:** Only score from live page content. Never use AI training data as a substitute for actual page content. If a page can't be fetched, show "Blocked" — not a guess.

## User Flow (6 Steps)

### Step 1 — Discover
User enters **Manufacturer Name** (required) plus one of:
- **Product Category** → Claude discovers top 5 parts, user selects one in Step 2
- **Part Number** → Skips part discovery, goes straight to distributor search
- **Direct Product URL** → Fetches page, extracts MPN/name, goes to distributor search

### Step 2 — Select Part (category path only)
Shows Claude-discovered parts. User clicks one to proceed.

### Step 3 — Agentic Discoverability
Distributors discovered via **Google search** (Serper.dev), not Claude inference.
- Searches for `"MPN" "manufacturer" buy`, `"MPN" distributor`, `"MPN" manufacturer price availability`, etc.
- Filters out aggregators (Octopart, FindChips, etc.), non-distributor sites, non-North-America TLDs
- Classifies distributors via Claude (proper names, relationship type)
- Shows which distributors have confirmed PDPs vs. search pages
- User selects 1–10 distributors to audit

### Step 4 — Confirm URLs
For each selected distributor + manufacturer:
- If search-first already found a PDP URL → pre-filled, marked "PDP Found"
- If not → resolves via additional SerpAPI/Serper queries with Claude URL selection
- User can manually paste/edit any URL
- "Open" button to verify each URL in a new tab

### Step 5 — Audit
For each site (manufacturer + selected distributors):
1. Fetch page content via `/api/fetch-page` (Brightdata → Jina fallback)
2. Validate it's a real PDP (not a search page, CAPTCHA, or JS shell)
3. Score 13-15 fields via Claude (shared fields for MFR, all fields for distributors)
4. If fetch fails → "Blocked" card, no score

### Step 6 — Results
Three tabs:
- **Gap Report** — Cards per site showing score, top gaps, source badge (Live/Blocked)
- **Full Matrix** — Field-by-field comparison grid
- **Agentic Discoverability** — Which distributors are agentic-ready

Re-audit available per blocked card (user pastes working URL).
CSV export available.

## Server Routes (`server.js`)

| Route | Method | Purpose |
|---|---|---|
| `/api/claude` | POST | Proxy to Anthropic Messages API |
| `/api/search` | POST | Proxy to Serper.dev Google search (with 1-hour in-memory cache) |
| `/api/fetch-page` | POST | Fetches page content: Brightdata (with JS rendering + Cloudflare detection) → Jina AI fallback |
| `*` | GET | Serves built React frontend |

### Page Fetching Pipeline (`/api/fetch-page`)
1. **Brightdata Web Unlocker** (if `BRIGHTDATA_API_KEY` set)
   - `render_js: true` for JS-rendered SPAs
   - Extra options for CF-heavy sites (digikey.com, newark.com, farnell.com, element14.com)
   - Cloudflare challenge detection (`isBlockedPage()`)
   - HTML stripped to plain text via `htmlToText()`
   - Returns up to 15,000 chars
2. **Jina AI fallback** (no key needed)
   - Returns up to 8,000 chars
   - Detects access denied / Cloudflare pages
3. If both fail → 502 error

## Scoring Model

### Fields (per site)
**Shared (manufacturer + distributor):** Product Name, Part Number, Images, Description, Specifications, Dimensions, Datasheets/Docs, Videos, Category/Breadcrumb, SEO/Keywords, Cross References, Compliance, Related Parts

**Distributor only:** Price/Pricing Tier, Availability/Lead Time

### Scoring
- `high` = present and complete on live page
- `medium` = partially present
- `low` = missing or not found
- `overallScore` = weighted average (high=100, medium=50, low=0)

### Agentic Readiness
- `✓ Agentic-Ready` = PDP URL directly discoverable via Google search for the MPN
- `⚠ Not Agentic-Ready` = Only search/category pages found (AI procurement agents can't reach the product)

## Known Issues & Limitations

### Critical
1. **Blocked distributor sites** — Digi-Key, Newark, Farnell, and some others still return Cloudflare challenge pages even through Brightdata Web Unlocker. The `render_js: true` flag should help but may not solve all cases. May need Brightdata Browser API (full Chrome) for the hardest sites.

2. **URL resolution gaps** — Serper.dev searches don't always find PDPs for niche or less-indexed parts. The fallback is a search-page URL, which will be marked "Blocked" or "Unauditable" during the audit.

### Moderate
3. **Single-file frontend** — `App.jsx` is 1,355 lines. Manageable but getting heavy. If adding significant features, consider splitting into components.

4. **No authentication** — Anyone with the URL can run audits and burn API credits. Fine for demo stage; needs auth before scaling.

5. **No rate limiting** — Each audit can trigger 10+ Claude calls + 10+ Serper queries + 10+ Brightdata fetches. A single user hammering the tool could burn significant API budget.

### Minor
6. **Stale env vars** — `FIRECRAWL_API_KEY` and `SERPAPI_KEY` can be removed from Render. They're unused but still set.
